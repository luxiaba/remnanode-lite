# Справочник по конфигурации

> Это русский перевод. Если настройки или правила расходятся с [английским справочником](../../configuration.md), используйте английский оригинал.

[К индексу документации на русском](README.md)

Здесь перечислены все поддерживаемые параметры и указано, откуда они читаются. Настройки Node, переменные Docker Compose и параметры установщика относятся к разным частям системы, даже если называются похоже.

## Источники и приоритет

При запуске Node выбирает файл конфигурации в следующем порядке:

1. Путь, указанный переменной `REMNANODE_ENV` в окружении запуска.
2. `/etc/remnanode/node.env`, если этот файл существует.
3. `.env` в текущем рабочем каталоге.

Сначала Node читает файл, а затем применяет известные непустые значения из окружения процесса. Пустая переменная окружения не стирает значение из файла. Если заданы оба параметра Secret, `SECRET_KEY` имеет приоритет над `SECRET_KEY_FILE`.

systemd и OpenRC указывают путь `/etc/remnanode/node.env` через `REMNANODE_ENV`, но не исполняют и не экспортируют файл целиком. Node читает его как данные, поэтому неизвестные ключи и Secret не попадают в окружение процесса. Docker Compose передаёт выбранные переменные непосредственно в контейнер.

После изменения конфигурации перезапустите Node или пересоздайте контейнер. Горячая перезагрузка конфигурации не поддерживается.

## Конфигурация времени выполнения Node

Процесс Go непосредственно читает параметры из этой таблицы. Пути по умолчанию соответствуют стандартной структуре контейнерного образа проекта и нативных установщиков.

| Переменная | Обязательна | Значение по умолчанию | Назначение и ограничения |
| --- | --- | --- | --- |
| `NODE_PORT` | Да | Нет | HTTPS-порт, через который Panel подключается к Node. Все способы запуска принимают только `1..65535`; недопустимое значение приводит к ошибке до открытия слушателя. |
| `NODE_BIND_ADDR` | Нет | Пусто | Адрес прослушивания HTTPS. Пустое значение означает все локальные адреса. На многосетевом хосте укажите конкретный адрес, доступный для Panel. |
| `SECRET_KEY` | При определённых условиях | Пусто | Полный Secret Key, выданный Panel. Непустое значение имеет приоритет над `SECRET_KEY_FILE`. |
| `SECRET_KEY_FILE` | При определённых условиях | Пусто | Читать Secret из обычного файла. В нативной установке используется `/etc/remnanode/secret.key`. |
| `XRAY_BIN` | Нет | `/usr/local/lib/remnanode/rw-core` | Путь к исполняемому файлу rw-core. |
| `GEO_DIR` | Нет | `/usr/local/share/remnanode/xray` | Каталог с `geoip.dat`, `geosite.dat` и необязательными данными zapret. |
| `LOG_DIR` | Нет | `/var/log/remnanode` | Каталог журналов stdout и stderr rw-core. |
| `ASN_DB_PATH` | Нет | `/usr/local/share/remnanode/asn/asn-prefixes.bin` | Компактная база ASN. Если она недоступна, общий список плагина `asList` становится пустым, а остальные основные функции продолжают работать. |
| `INTERNAL_SOCKET_PATH` | Нет | `/run/remnanode/internal.sock` | Unix-сокет между Node и rw-core. В производственной среде обычно менять его не следует. |
| `INTERNAL_REST_TOKEN` | Нет | Случайное значение при каждом запуске | Токен внутренней конфигурации и webhook. Безопаснее оставить пустым; фиксированное значение предназначено главным образом для контролируемой отладки. |
| `DISABLE_HASHED_SET_CHECK` | Нет | `false` | При true Node перестаёт использовать хеш конфигурации для пропуска неизменившегося запуска; каждый запрос start перезапускает core. Только для отладки. |
| `LOW_MEMORY` | Нет | `false` | Включает политику малого объёма памяти. Производственный Compose включает её по умолчанию. Нативные установщики включают её автоматически, если общий объём памяти машины не превышает 512 MiB. |
| `BODY_LIMIT_MB` | Нет | `0` (автоматически) | Дополнительный предел тела запроса для публичного HTTPS-сервера `/node`, допустимый диапазон `0..1024`. В режиме малого объёма памяти явное значение не может превышать 16. Внутренний Unix webhook всегда ограничен 8 KiB. |
| `GOMEMLIMIT` | Нет | Пусто | Мягкий предел памяти, управляемой средой выполнения Go. Принимает число байт, `B/KiB/MiB/GiB/TiB` или `off`. Явное значение переопределяет значение режима малого объёма памяти. |
| `NODE_CONTRACT_VERSION` | Нет | Версия контракта, заданная при сборке | Переопределяет `nodeVersion`, сообщаемый Panel. Используйте только для отладки контракта или экстренной проверки совместимости. |
| `XRAY_CORE_VERSION` | Нет | Определяется по фактическому бинарному файлу | Переопределяет сообщаемую версию rw-core. Не устанавливает, не обновляет и не проверяет соответствующий бинарный файл. |

Логические значения не зависят от регистра и принимают `true/false`, `1/0` или `yes/no`. Недопустимые логические, числовые значения или строки версии приводят к завершению Node до начала прослушивания, а не к молчаливому откату.

### Режим малого объёма памяти

`LOW_MEMORY=1` одновременно изменяет следующие границы времени выполнения:

| Граница | Обычный режим | Режим малого объёма памяти |
| --- | ---: | ---: |
| Мягкий предел памяти Go | Политика Go по умолчанию | 180 MiB |
| Предел TCP-соединений Node API | 128 | 16 |
| Активные HTTP-обработчики | 32 | 4 |
| Автоматический предел тела запроса | 256 MiB | 16 MiB |
| Ожидание готовности rw-core | 20 секунд | 90 секунд |

Эти значения не являются жёсткими пределами контейнера или cgroup. `GOMEMLIMIT` ограничивает память, управляемую средой выполнения Go; это не ограничение только heap и не ограничение полного RSS процесса. rw-core, память процесса Go вне runtime, tmpfs и прочая память вместе учитываются в общем пределе 448 MiB для Compose/systemd/OpenRC.

Для публичных маршрутов также действуют меньшие индивидуальные пределы. Даже при более высоком `BODY_LIMIT_MB` эффективный предел любого текущего публичного запроса не превышает 16 MiB.

### Secret Key

Secret представляет собой JSON в кодировке base64 или base64url, с padding или без него; размер закодированного значения не должен превышать 256 KiB. После декодирования должны присутствовать:

```text
caCertPem
jwtPublicKey
nodeCertPem
nodeKeyPem
```

Для нативной установки храните его в отдельном файле с ограниченными правами:

```bash
sudo install -d -o root -g remnanode -m 0750 /etc/remnanode
printf '%s' 'PASTE_THE_COMPLETE_SECRET_KEY' \
  | sudo tee /etc/remnanode/secret.key >/dev/null
sudo chown root:remnanode /etc/remnanode/secret.key
sudo chmod 0640 /etc/remnanode/secret.key
```

Файл Secret должен быть обычным файлом, а не символической ссылкой. Содержимое может не иметь завершающего перевода строки, иметь один LF или один CRLF. Внутренние пробельные символы отклоняются.

Для однофайлового развёртывания Docker используйте отображение YAML (mapping):

```yaml
environment:
  NODE_PORT: "38329"
  SECRET_KEY: "PASTE_THE_COMPLETE_SECRET_KEY"
  LOW_MEMORY: "1"
```

Не используйте форму списка `- SECRET_KEY="..."`. В ней кавычки становятся частью значения и вызывают ошибку декодирования base64. Установите режим `0600` для файла Compose или `.env`, содержащего Secret, и никогда не добавляйте его в Git.

## Синтаксис и пределы `node.env`

`node.env` использует ограниченный синтаксис dotenv. Это не shell-скрипт:

```env
NODE_PORT=2222
SECRET_KEY=
SECRET_KEY_FILE=/etc/remnanode/secret.key
LOW_MEMORY=1
BODY_LIMIT_MB=
```

Правила разбора:

- Допускаются пустые строки, комментарии, начинающиеся с `#`, и необязательный префикс `export KEY=value`.
- Значения могут быть без кавычек или заключены в одну согласованную пару одинарных либо двойных кавычек.
- Команды, подстановка переменных и shell substitution никогда не выполняются.
- Если ключ встречается несколько раз, действует последнее значение. Установщик объединяет дублирующиеся ключи, которыми он управляет.
- Файл ограничен 1 MiB, 4096 строками и 256 присваиваниями.
- В Linux файл открывается с `O_NOFOLLOW|O_NONBLOCK|O_CLOEXEC`, а его состояние до и после чтения сравнивается через тот же файловый дескриптор.
- Неизвестные ключи учитываются в пределах файла, но не входят в конфигурацию Node и не передаются автоматически в rw-core.

Стандартный нативный файл принадлежит `root:remnanode` и имеет режим `0640`.

### Пример нативной конфигурации для малого объёма памяти

```env
NODE_PORT=2222
NODE_BIND_ADDR=
SECRET_KEY=
SECRET_KEY_FILE=/etc/remnanode/secret.key

XRAY_BIN=/usr/local/lib/remnanode/rw-core
GEO_DIR=/usr/local/share/remnanode/xray
LOG_DIR=/var/log/remnanode
ASN_DB_PATH=/usr/local/share/remnanode/asn/asn-prefixes.bin
INTERNAL_SOCKET_PATH=/run/remnanode/internal.sock
INTERNAL_REST_TOKEN=

DISABLE_HASHED_SET_CHECK=false
LOW_MEMORY=1
BODY_LIMIT_MB=
```

Обычно добавлять `GOMEMLIMIT=180MiB` не нужно: `LOW_MEMORY=1` уже задаёт тот же мягкий предел Go по умолчанию. Переопределяйте его только после измерения ресурсов.

Поддерживаемый нативный шаблон находится в [`deploy/node.env.example`](../../../deploy/node.env.example).

## Подстановка Docker Compose

Файл `.env.example` в корне репозитория обрабатывается CLI Compose. Это не контейнерный `node.env`, и процесс Go не обнаруживает его самостоятельно.

| Переменная | Потребитель | Описание |
| --- | --- | --- |
| `REMNANODE_IMAGE` | Compose | Тег образа или `name@sha256:...`; в Node не передаётся. |
| `NODE_PORT` | Compose -> Node | Передаётся в контейнер во время выполнения. |
| `NODE_BIND_ADDR` | Compose -> Node | Передаётся в контейнер во время выполнения. |
| `SECRET_KEY` | Compose -> Node | Передаётся через окружение контейнера и виден в локальных метаданных Docker. |
| `LOW_MEMORY` | Compose -> Node | В производственных шаблонах по умолчанию `1`. |
| `DISABLE_HASHED_SET_CHECK` | Compose -> Node | В производственных шаблонах по умолчанию `false`. |
| `BODY_LIMIT_MB` | Compose -> Node | Оставьте пустым для значения режима малого объёма памяти по умолчанию. |
| `GOMEMLIMIT` | Compose -> Node | Оставьте пустым, чтобы значение по умолчанию выбрал `LOW_MEMORY`. |

Для большого числа независимых узлов файл `.env` не нужен. Значения времени выполнения можно записать непосредственно в отображение `environment` Compose. Начните с [`deploy/compose.single-file.yaml`](../../../deploy/compose.single-file.yaml) и следуйте [руководству по развёртыванию Docker Compose](deployment-docker.md).

`latest` влияет только на то, какой образ будет выбран при следующем pull. Уже работающий контейнер он не заменяет:

```bash
docker compose pull
docker compose up -d --no-build --force-recreate
```

## Конфигурация установщика и средства обновления

Следующие переменные обрабатывают shell-скрипты; это не параметры времени выполнения daemon. Некоторые параметры артефактов можно сохранить в `node.env` для последующего запуска `install-xray.sh`.

| Переменная или параметр | Потребитель | Назначение |
| --- | --- | --- |
| `RNL_REPO` | Все скрипты установки | Репозиторий-источник Release; по умолчанию `luxiaba/remnanode-lite`. |
| `RNL_TAG` | Установка, обновление, удаление | Точный тег, например `vX.Y.Z` или `vX.Y.Z-rnl.N`. |
| `RNL_INSTALL_XRAY=0` / `--skip-xray` | Установка | Пропустить rw-core при новой установке. Не рекомендуется для обычной производственной установки. |
| `RNL_UPGRADE_XRAY=1` / `--upgrade-xray` | Обновление | Одновременно обновить rw-core, geo и ASN. По умолчанию установленные ресурсы rw-core сохраняются. |
| `RNL_INSTALL_ASN=0` | `install-xray` | Пропустить базу ASN; `asList` становится пустым списком. |
| `RNL_TMP_ROOT` | Установщик | Расширенная настройка корневого каталога транзакции; по умолчанию `/var/lib/remnanode-installer`, доступного только root. |
| `CUSTOM_CORE_URL` | `install-xray` | Пользовательский HTTPS URL бинарного файла core для Linux. |
| `CUSTOM_CORE_SHA256` | `install-xray` | Обязательная SHA-256 для пользовательского core. |
| `ASN_DB_URL` | `install-xray` | Пользовательский HTTPS URL для RWASNDB. |
| `ASN_DB_SHA256` | `install-xray` | Обязательная SHA-256 для пользовательской базы ASN. |
| `GEO_ZAPRET_FILE` | Установка | Атомарно скопировать локальный файл как `geo-zapret.dat`. |
| `IP_ZAPRET_FILE` | Установка | Атомарно скопировать локальный файл как `ip-zapret.dat`. |
| `XRAY_CORE_VERSION` / `--version` | `install-xray` | Выбрать Release rw-core. Для версии, отличной от закреплённой проектом, также требуется SHA-256. |
| `XRAY_CORE_SHA256` / `--sha256` | `install-xray` | Обязательный digest для Release rw-core, не закреплённого проектом. |

`RNL_ENSURE_SERVICE_STARTED`, `RNL_ENSURE_SERVICE_ENABLED` и `RNL_EXTERNAL_ASSET_ROLLBACK` относятся к внутреннему протоколу транзакций установщика. Не задавайте их вручную.

Пользовательский core всё равно использует geo-данные из выбранного Release rw-core, после чего бинарный файл core заменяется проверенным пользовательским файлом. Для каждого пользовательского URL требуется соответствующая SHA-256; отсутствие digest приводит к ошибке до записи любого целевого пути.

Установка и обновление ведут себя по-разному. Повторный запуск `install-node*.sh --install` для полной установки обновляет rw-core, geo и ASN из выбранного Release. Явный `--upgrade` и прямой запуск `upgrade.sh` сохраняют установленные ресурсы, если не указаны `--upgrade-xray` или `RNL_UPGRADE_XRAY=1`.

## Переопределение версий

Версия проекта, версия официального контракта и версия rw-core являются отдельными понятиями:

- Версия проекта идентифицирует Releases, бинарные файлы и теги образов.
- Версия контракта обозначает базовую версию официального Node API, которая фактически реализована и сообщается Panel.
- Версия rw-core обозначает core, включённый в поставку или установленный в системе.

Не выводите версию контракта из версии проекта и не используйте переопределение, чтобы заявить совместимость до её реализации. Посмотреть декларацию текущего бинарного файла можно командой:

```bash
remnanode-lite version
```
