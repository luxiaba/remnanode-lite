# Политика безопасности

> English is the only authoritative source. If this translation differs from the [canonical English security policy](../../../SECURITY.md), follow the English document.

[К русскому индексу документации](README.md)

В этом документе описано, как сообщать об уязвимостях в Remnanode Lite, какие
выпуски получают поддержку безопасности и какие границы доверия должны
учитывать операторы. Рекомендации по усилению защиты при развёртывании и детали
реализации приведены в английском документе [«Architecture»](../../architecture.md) и
русском [руководстве по эксплуатации](operations.md).

## Сообщение об уязвимости

Сообщайте о проблемах безопасности через
[закрытую форму GitHub для сообщений об уязвимостях](https://github.com/luxiaba/remnanode-lite/security/advisories/new).
Не публикуйте в открытых issue, discussions, журналах или снимках экрана ничего
из следующего:

- `SECRET_KEY`, JWT, центры сертификации, сертификаты узла или закрытые ключи.
- URL Panel, реальные IP-адреса, имена хостов и другие сведения, позволяющие
  идентифицировать узел.
- Запросы, ответы, конфигурацию или журналы времени выполнения без удаления
  конфиденциальных данных.
- Полные инструкции по эксплуатации уязвимости, которые позволили бы провести
  атаку до выхода исправления.

По возможности укажите затронутую версию или commit, способ развёртывания,
ожидаемые последствия, минимальный сценарий воспроизведения и известные меры
смягчения риска. Используйте для воспроизведения вымышленные адреса и очищенные
данные. При необходимости сопровождающие запросят дополнительные сведения в
закрытом advisory.

Если секрет мог быть раскрыт, немедленно замените его. Удаление значения в
последующем commit не удаляет его из истории Git, журналов, кэшей, реестров и
других копий.

## Поддерживаемые версии

Первый финальный выпуск ещё не опубликован. До его выхода `edge`, `sha-*` и
`candidate-sha-*` являются кандидатными сборками, для которых не предусмотрены
обязательства по долгосрочной поддержке безопасности.

После первого финального выпуска действует следующая политика:

| Версия | Поддержка безопасности |
| --- | --- |
| Стабильный выпуск, на который указывает `latest` | Получает исправления безопасности |
| Предыдущий стабильный выпуск в той же линии выпусков | Проблемы с высоким уровнем воздействия исправляются, если обновление или откат остаются практически осуществимыми |
| `edge`, исторические кандидатные сборки и более старые выпуски | Исправления не гарантируются; необходимо обновиться до поддерживаемого выпуска |

Соответствующий GitHub Security Advisory и примечания к выпуску могут сузить
или расширить поддержку для конкретной проблемы.

## Границы доверия во время выполнения

Remnanode Lite — узел управления сетью, а не обычный непривилегированный
веб-сервис:

- Публичный API Panel-to-Node требует TLS 1.3 или новее, взаимную TLS-аутентификацию
  и Bearer JWT с алгоритмом RS256.
- Docker использует сетевое пространство имён хоста. Возможность `NET_ADMIN`
  позволяет контейнеру управлять таблицами nftables этого проекта и разрывать
  подходящие соединения через `NETLINK_SOCK_DIAG`; `NET_BIND_SERVICE` позволяет
  слушать привилегированные порты.
- В текущем образе контейнер запускается от имени root, сбрасывает все остальные
  capabilities, включает `no-new-privileges` и использует корневую файловую
  систему только для чтения. Тем не менее сеть хоста и `NET_ADMIN` остаются
  явной границей доверия к хосту.
- Запускайте только проверенные образы, созданные этим репозиторием. В
  production фиксируйте точную версию или digest манифеста и проверяйте
  аттестацию сборки.
- Node не сохраняет полную конфигурацию Xray, полученную от Panel. После
  перезапуска Panel синхронизирует её заново. Журналы времени выполнения также
  могут храниться только во временном хранилище ограниченного размера.

Поддерживаемый production-шаблон в одном файле находится в
[`deploy/compose.single-file.yaml`](../../../deploy/compose.single-file.yaml).
Сохраняйте заданные в нём ограничения capabilities, файловой системы только
для чтения, tmpfs, процессов, памяти, CPU, healthcheck и ротации журналов, если
только проверенное изменение развёртывания явно не заменяет их.

## Работа с секретами

При нативном развёртывании через systemd или OpenRC храните секрет в
`/etc/remnanode/secret.key` с владельцем и режимом доступа `root:remnanode 0640`.
Go-процесс читает конфигурацию и секреты через ограниченные пути к файлам, не
допускающие символических ссылок; полное содержимое `node.env` не экспортируется
в окружение службы.

При однофайловом развёртывании с Compose секрет неизбежно хранится внутри файла.
Поэтому он виден в метаданных контейнера, доступных через `docker inspect`.
Защитите файл следующей командой:

```bash
chmod 600 docker-compose.yaml
```

Кроме того, ограничьте доступ к Docker socket, резервным копиям, истории shell
и учётным записям администраторов хоста. Перед запуском rw-core Node удаляет
`SECRET_KEY`, `SECRET_KEY_FILE`, `INTERNAL_REST_TOKEN` и `REMNANODE_ENV` из
унаследованного окружения. Он подменяет управляемые пути к ресурсам и внутренний
токен webhook. По умолчанию этот токен генерируется случайным образом при каждом
запуске; явное значение принимается только после разбора конфигурации Go.
Остальные неуправляемые переменные окружения по-прежнему наследуются, поэтому
не передавайте в контейнер или нативную службу Node посторонние секреты.

Никогда не добавляйте в Git `.env`, развёрнутый файл Compose с настоящим
секретом, `/etc/remnanode/node.env`, `secret.key`, сертификаты, закрытые ключи
или необработанные материалы приёмочных испытаний.

## Контроль цепочки поставки

В репозитории в настоящее время применяются следующие меры:

- GitHub Actions зафиксированы по полным SHA commit.
- Go modules проверяются, а запланированный
  [workflow безопасности](../../../.github/workflows/security.yml) запускает
  `govulncheck` для достижимого Go-кода.
- Базовые образы контейнеров зафиксированы по digest манифеста.
- Источники rw-core, geo-данных и ASN зафиксированы по версии или commit и
  проверяются по digest загружаемых файлов.
- Образы выпусков содержат SBOM, provenance BuildKit и аттестацию сборки GitHub.
- Бинарные файлы выпуска, ресурс Compose и ресурсы данных покрыты файлом
  `SHA256SUMS`.

Эти меры не делают сборки воспроизводимыми байт в байт. Пакеты Debian,
устанавливаемые Dockerfile, пока не зафиксированы по snapshot и точным версиям.
Идентифицируйте финальный артефакт одновременно по digest манифеста в реестре,
SBOM, provenance и аттестации; не доверяйте одному только имени tag.

CI репозитория определён в
[`.github/workflows/ci.yml`](../../../.github/workflows/ci.yml). В корневом
[`CHANGELOG.md`](../../../CHANGELOG.md) фиксируются видимые пользователям
изменения выпусков. Файлы в `docs/archive/`, например
[запись об устранении замечаний аудита 2026 года](../../archive/2026-07-audit-remediation.md),
сохраняют исторический контекст и не должны считаться описанием текущего
состояния безопасности или статуса выпуска.

## Принципы безопасного проектирования

- До любых побочных эффектов аутентифицировать внешние данные и применять к ним
  ограничения на распаковку, размер тела, декодирование JSON и контракт.
- Ограничивать процессы, очереди, тела запросов, параллелизм, вывод внешних
  команд и длительность завершения работы.
- Сохранять единственного владельца для rw-core, снимков плагинов и состояния
  nftables. Неудачный побочный эффект не должен фиксировать ложное локальное
  состояние успеха.
- Владеть только процессом rw-core этого проекта, внутренними socket и
  фиксированными таблицами nftables. Не брать под управление общую политику
  firewall хоста.
- Считать разрыв соединений операцией над сетью хоста. Она просматривает сетевое
  пространство имён хоста в поисках TCP-соединений, соответствующих целевому IP,
  и может закрыть соединение другого процесса, если тот использует тот же IP.
  Поэтому production-узлы должны быть выделенными средами выполнения сетевых
  операций. Путь через Panel фильтрует локальные и специальные адреса, но
  административная команда `remnanode-lite kill-sockets` вызывает адаптер ядра
  напрямую и не предоставляет такой защиты бизнес-уровня.
- Хранить данные приёмки выпуска в обезличенном виде. Ни evidence, ни исходные
  пакеты захвата не должны содержать данные, по которым можно восстановить
  сведения о пользователях или production-средах.

## Изменения, связанные с безопасностью

Изменения, влияющие на безопасность, должны соответствовать
[CONTRIBUTING.md](../../../CONTRIBUTING.md), включать регрессионные тесты и
проходить проверки, подходящие для затронутой границы. Полный локальный gate:

```bash
REMNANODE_OFFICIAL_SOURCE=/absolute/path/to/pinned-official-source \
REQUIRE_GOVULNCHECK=1 \
  bash scripts/check.sh
```

Успешное прохождение этого gate подтверждает только проверки на уровне
репозитория. Оно не заменяет реальные тесты пространств имён Linux, проверку
аттестации кандидата, интеграцию с Panel, тесты отказов при ограниченных
ресурсах или длительную приёмку M8 там, где они требуются.
