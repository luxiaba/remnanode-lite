# Документация Remnanode Lite

Английский [индекс документации](../../README.md) является единственным нормативным источником. Русский перевод предназначен для удобства операторов и может обновляться с задержкой. Правила перевода описаны в [политике локализации, English](../README.md).

[Главная страница на русском](../../../README.ru.md) · [English](../../../README.md) · [简体中文](../../../README.zh-CN.md)

Этот раздел помогает развернуть и обслуживать Remnanode Lite. Документы для разработчиков, сопровождения контракта, приемки и выпуска остаются на английском; ссылки на них явно отмечены как English.

Если перевод расходится с кодом, Release-артефактом, результатом выполнения или английской документацией, следуйте английскому источнику и сообщите о расхождении.

## Быстрый маршрут для оператора

### Развернуть новый узел

1. Прочитайте [обзор проекта](project.md), чтобы проверить границы поддержки и нецели.
2. Выберите [Docker Compose](deployment-docker.md) либо [нативную установку systemd/OpenRC](deployment-native.md).
3. Заполните параметры и Secret по [справочнику конфигурации](configuration.md).
4. Перед выбором образа изучите [политику версий и тегов](versioning.md), особенно различия между `latest`, точной версией, `edge`, `sha-*` и `candidate-sha-*`.
5. После запуска проверьте health, связь с Panel и журналы rw-core по [руководству по эксплуатации](operations.md).

На полном хосте `512 MiB RAM / 1 vCPU / 2 GB disk` сохраняйте заданные проектом лимиты памяти, CPU, PID, tmpfs и журналов. Не собирайте исходный код на таком production-узле.

### Обслуживать работающие узлы

1. Начните с healthcheck, журналов, обновления, отката и диагностики в [руководстве по эксплуатации](operations.md).
2. Проверьте источник и приоритет параметров в [справочнике конфигурации](configuration.md).
3. Для памяти, диска, журналов и времени остановки используйте [бюджет ресурсов, English](../../development/resource-budget.md).
4. При проблемах протокола или жизненного цикла смотрите [архитектуру, English](../../architecture.md) и [контракт 2.8.0, English](../../development/contract-2.8.0.md).
5. Для отката используйте ранее записанную точную версию или manifest digest. Не используйте `edge` и не полагайтесь на прежнее значение `latest`.

## Ключевые понятия

### Версия проекта не равна версии контракта

`Version` идентифицирует сборку Remnanode Lite. `ContractVersion` обозначает поведение официального Node, которое фактически реализовано и сообщается Panel. Проект может заранее начать линию `rnl.N`, но не может заявить незавершенный официальный контракт. Подробности приведены в [политике версий](versioning.md).

### Кандидатный образ не является Release

`edge`, автоматический `sha-*` и ручной `candidate-sha-*` собираются из `main` для наблюдения и серверной приемки. Формальный Release существует только при наличии Git tag, GitHub Release и точного GHCR version tag, неизменяемых по политике проекта. Для технической фиксации используйте digest мультиархитектурного manifest. Строка версии в исходном коде не доказывает публикацию.

### Вывод о совместимости всегда ограничен

Статические contract tests, успешное подключение к Panel, длительный ресурсный тест и приемка дистрибутива доказывают разные свойства. Не заменяйте одним результатом реализацию, автоматическую проверку, приемку среды и факт публикации одновременно.

## Термины

| Термин | Значение |
| --- | --- |
| Node | Долгоживущий процесс `remnanode-lite`, принимающий запросы Panel и владеющий жизненным циклом rw-core |
| rw-core | Бинарный Xray Core, обслуживающий proxy traffic и управляемый Node |
| `Version` | Идентичность сборки проекта, GitHub Release и точного тега образа |
| `ContractVersion` | Реализованный официальный behavioral baseline, по умолчанию сообщаемый Panel |
| operation epoch | Возрастающий token владения одной операцией Xray start/stop, не идентичность процесса |
| process lease | Краткоживущий допуск, привязанный к process epoch rw-core и abstract socket, не позволяющий mutation перейти между процессами |
| lifecycle lease | Shared/exclusive координация HTTP для start, stop, plugin/user mutation и reset-capable stats; не lock-файл |
| кандидат `C` | Commit в `main`, замороженный для приемки в реальной среде |
| финальный commit `F` | Единственный разрешенный commit с Release-документами после `C`; формальный Git tag указывает на `F` |
| manifest digest | Контентный адрес `sha256:...` мультиархитектурного GHCR index, надежнее перемещаемого тега |

## Сопровождение перевода

- Сначала обновляется английский canonical-документ, затем перевод.
- Команды, пути, имена переменных, числовые лимиты и support claims не локализуются по смыслу.
- Реальные Secrets, JWT, сертификаты, адреса хостов и пользовательские данные запрещены в примерах.
- Исторические аудиты и roadmap не заменяют код и автоматизацию как источник текущего поведения.
- При расхождении всегда следуйте английскому источнику и обновите metadata SHA-256 по [политике локализации, English](../README.md).
